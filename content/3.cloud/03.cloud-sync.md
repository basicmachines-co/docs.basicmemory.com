---
title: Cloud Sync
description: Keep a local Markdown folder synchronized with a Basic Memory Cloud project.
---

Cloud Sync is an optional workflow for people who want to edit Cloud projects as local Markdown files in tools such as Obsidian, VS Code, or another text editor.

You do not need Cloud Sync to use Basic Memory Cloud. The web app and hosted MCP endpoint work without a local copy.

::note
This is an advanced hybrid workflow. For the shortest command-line overview, see [Cloud CLI](/cloud/cloud-cli). For the shortest practical sync setup, see [Edit Locally and in the App](/cloud/edit-locally-and-in-the-app).
::

---

## Choose the right workflow

| What you want | Recommended approach |
|---|---|
| Work entirely in the browser and AI tools | Use the [web app](/cloud/web-app) and hosted MCP endpoint |
| Import a folder once | Upload a Project ZIP from **Settings → Import** |
| Keep a personal Cloud project and local folder synchronized | Use two-way `bisync` |
| Publish a local folder to a team workspace | Use one-way `sync` |
| Keep sensitive projects only on your computer | Use the [open-source local installation](/local/local-install) |

::warning
Two-way `bisync` is supported only for **personal workspaces**. For a project in a shared team workspace, use one-way `bm cloud sync` to publish local changes to Cloud.
::

---

## Before you start

You need:

- A Basic Memory Cloud account
- The current Basic Memory CLI
- A Cloud project
- A local directory for its Markdown files

Check your installed version:

```bash
bm --version
```

Update an older installation before following this guide:

```bash
bm update
```

---

## Set up a personal project for two-way sync

### 1. Sign in

```bash
bm cloud login
```

This authenticates the CLI with Basic Memory Cloud. It does not move or sync any files.

### 2. Configure file synchronization

```bash
bm cloud setup
```

Basic Memory installs or configures `rclone`, which performs the file transfer.

### 3. Attach a local directory

For a new Cloud project:

```bash
bm project add research --cloud --local-path ~/Documents/research
```

For an existing Cloud project:

```bash
bm cloud sync-setup research ~/Documents/research
```

The local directory can live anywhere on your computer.

### 4. Preview the initial sync

The first two-way sync needs a baseline. Preview it before changing files:

```bash
bm cloud bisync --name research --resync --dry-run
```

Review the proposed uploads, downloads, and deletions carefully.

### 5. Establish the baseline

```bash
bm cloud bisync --name research --resync
```

After the baseline exists, do not routinely use `--resync`.

### 6. Sync later changes

```bash
bm cloud bisync --name research
```

Changes made locally or through the web app are copied to the other side. When the same file changed in both places, the newer version wins.

## Publish a local project to a team workspace

Two-way bisync is intentionally unavailable for organization workspaces. To make Cloud match a local folder, configure the project and run one-way sync:

```bash
bm cloud sync-setup team-wiki ~/Documents/team-wiki
bm cloud sync --name team-wiki --dry-run
bm cloud sync --name team-wiki
```

::warning
One-way sync treats the local folder as the source of truth. Cloud-only file changes can be replaced or removed. Always run `--dry-run` first.
::

For normal collaborative editing, use the web app and MCP tools instead. See [Collaboration](/teams/about).

## Check sync status

Check authentication, Cloud health, and configured project sync:

```bash
bm cloud status
```

Verify that the local and Cloud files match without transferring data:

```bash
bm cloud check --name research
```

Use a faster one-direction check when you only need to detect files missing from the destination:

```bash
bm cloud check --name research --one-way
```

## Work safely

- Run `--dry-run` before the first sync or any operation that may move or delete many files.
- Allow one sync operation to finish before starting another.
- Create a [Cloud Snapshot](/cloud/cloud-snapshots) before a major reorganization.
- Do not edit the same file locally and in the web app at the same time.
- Keep at least one Markdown file in a new sync directory before establishing its first baseline.

Basic Memory respects `.bmignore` and project `.gitignore` patterns during file synchronization. Use them to exclude generated files, dependency folders, credentials, and editor metadata.

## Troubleshooting

### Authentication failed

```bash
bm cloud logout
bm cloud login
```

### Project has no local sync path

Attach a local directory, then establish a baseline:

```bash
bm cloud sync-setup research ~/Documents/research
bm cloud bisync --name research --resync
```

### Bisync state is inconsistent

Clear only the synchronization metadata, preview a new baseline, and then apply it:

```bash
bm cloud bisync-reset research
bm cloud bisync --name research --resync --dry-run
bm cloud bisync --name research --resync
```

`bisync-reset` does not delete your project files.

### Too many deletions

Bisync stops when a change exceeds its deletion safety limit. Inspect the proposed operation:

```bash
bm cloud bisync --name research --dry-run
```

Do not force a new baseline until you understand why the files would be removed.

### First sync reports an empty directory

Add a starter Markdown file, then retry:

```bash
printf '# Research\n' > ~/Documents/research/README.md
bm cloud bisync --name research --resync
```

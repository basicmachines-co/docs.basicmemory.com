Basic Memory can run locally, in Basic Memory Cloud, or in a hybrid setup. Routing is the local CLI/MCP feature that decides whether a project is handled on your computer or forwarded to Basic Memory Cloud.

::note{icon="i-lucide-info"}
You do not need routing to use the Basic Memory web app or the hosted MCP endpoint at `https://cloud.basicmemory.com/mcp`. This page is for people who also use the Basic Memory CLI or a local MCP server and want selected projects to use Cloud.

For the shorter command-line overview, start with [Cloud CLI](/cloud/cloud-cli).
::

---

## How Routing Works

When Basic Memory runs on your computer, project-scoped requests follow this order:

1. **Explicit command flag** — Commands that support `--local` or `--cloud` use that flag for the current command.
2. **Project mode** — Projects marked with `bm project set-cloud` route through Cloud. Projects marked with `bm project set-local` stay local.
3. **Local default** — If no Cloud route is configured, Basic Memory uses the local project on your machine.

Cloud-routed projects use your saved API key first. If no API key is saved, Basic Memory can use your browser login token from `bm cloud login`.

---

## When To Use Routing

Use routing when you want one of these setups:

- Work in local tools while keeping a project in Basic Memory Cloud.
- Keep private projects local and route shared projects through Cloud.
- Use Cloud workspaces from scripts or terminal workflows.
- Move a project to Cloud without changing every project on your machine.

If you only use the web app and hosted MCP connection, skip this page.

---

## Authenticate The CLI

Cloud-routed projects need Cloud credentials. You can use either method.

```bash
# Interactive browser login
bm cloud login

# Or save an API key from the web app
bm cloud api-key save bmc_your_key_here
```

API keys are usually best for automation, shared terminals, and repeatable setup. See [API Keys](/cloud/api-keys) for creating and managing keys.

---

## Route A Project Through Cloud

Mark an existing local project as Cloud-routed:

```bash
bm project set-cloud research
```

After this, MCP tools and CLI commands for `research` route through Basic Memory Cloud.

If you have more than one Cloud workspace, attach the project to a specific workspace:

```bash
bm project set-cloud research --workspace team-name
```

You can pass a workspace name, slug, type, or workspace ID. To set a default workspace for future Cloud commands:

```bash
bm cloud workspace list
bm cloud workspace set-default team-name
```

::warning
`bm project set-cloud` changes routing; it does not delete your files. Basic Memory removes the local index entry and preserves the files on disk. If you want local files and Cloud content to stay in sync, see [Cloud Sync](/cloud/cloud-sync).
::

---

## Check Project Routing

```bash
bm project list
bm cloud status
```

`bm project list` shows each project's local path, cloud path, workspace, CLI route, and MCP transport. `bm cloud status` checks your Cloud connection and reports configured Cloud projects.

---

## Force One Command

Some CLI commands support `--local` and `--cloud` flags. Use them when you need a one-time override without changing the saved project mode.

The exact command depends on the operation. If a command supports routing flags, it will show them in its help output:

```bash
bm project info --help
```

---

## Return A Project To Local

To route a Cloud project back through your local files, give Basic Memory the local path:

```bash
bm project set-local research --local-path ~/Documents/research
```

If the project still has a previous local path recorded, you can omit `--local-path`:

```bash
bm project set-local research
```

Your Cloud data remains in Cloud. This only changes where local CLI and MCP requests go next.

## Common Setups

### Local Only

No setup required. Projects you add locally stay local unless you explicitly route them through Cloud.

### Cloud From Local Tools

Use Cloud as the source of truth for a selected project:

```bash
bm cloud api-key save bmc_your_key_here
bm project set-cloud research
```

### Hybrid

Keep personal projects local and route shared projects through Cloud:

```bash
bm cloud api-key save bmc_your_key_here

bm project set-cloud shared-notes
bm project set-cloud team-wiki --workspace acme
```

Local projects require no extra configuration.

### Gradual Migration

Upload or copy content into Cloud, then route the project through Cloud:

```bash
bm cloud upload ~/my-notes --project my-notes --create-project
bm cloud api-key save bmc_your_key_here
bm project set-cloud my-notes
```

Repeat for each project you want Cloud to own.

## Routing And Sync Are Different

Routing decides where Basic Memory handles a project.

Sync copies files between a local folder and Cloud storage.

You can route a project through Cloud without maintaining a local copy. You can also keep a local working folder synchronized with Cloud when that workflow is useful.

Start with [Edit Locally and in the App](/cloud/edit-locally-and-in-the-app) for the short version, or use [Cloud Sync](/cloud/cloud-sync) for the full reference.

---

## Credentials And Logout

`bm cloud login` signs the CLI into Basic Memory Cloud. It does not automatically move every local project into Cloud mode.

`bm cloud logout` removes the saved OAuth session and default workspace, but it does not remove a saved API key. If you saved an API key, Cloud-routed projects can still use it until you remove or replace it.

---

## How It Works Under the Hood

- **Local mode** uses the in-process Basic Memory service and your local project files.
- **Cloud mode** forwards project requests to Basic Memory Cloud using your API key or OAuth token.
- **Workspace routing** uses a per-project workspace when configured, then the default workspace, then an unambiguous single workspace when Basic Memory can resolve one.

The goal is simple: the same Basic Memory tools can work with local projects, Cloud projects, or a deliberate mix of both.
